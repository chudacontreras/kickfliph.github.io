Identificación del Proyecto

Dashboard.blackbox.org.ve es un sistema basado en web que permite al Administrador de Sistemas nonitorear, administrar y tomar deciciones sobre los servicios y servidores a su cargo. Este software trata de entregar al administrador la independencia de resolver problemas en el sitio donde se encuentre sin estar limitado por el tiempo ni el espacio siempre y cuando tenga acceso a internet.

Antecedentes

    • ¿Qué es un dashboard?

      Los dashboards son aplicaciones de business intelligence que le permiten a una organización visualizar la información más importante para monitorear, analizar y administrar el desempeño del negocio de manera más efectiva. Deben ser vistos más que como una pantalla con gráficas atractivas, como un sistema de información que ayuda a las organizaciones a optimizar su desempeño y alcanzar sus objetivos estratégicos. Un dashboard permite a los administradores monitorizar fácilmente los indicadores clave de rendimiento de los servicios.

      Las mejores prácticas en materia de desarrollo de dashboards indican que estas herramientas deben de contar con tres capas o vistas de información:

        • Capa o vista superior: es la capa donde los usuarios monitorean la información, la cual se presenta en forma resumida y usualmente gráfica, con el status de los indicadores más relevantes y las condiciones extraordinarias. Se utilizan alertas cuando el desempeño está fuera de los rangos establecidos.
        • Capa o vista intermedia: es la capa donde se despliegan los datos que soportan los indicadores y las alertas de la capa superior. Aquí, los usuarios pueden realizar cálculos complejos en los datos, moverse entre tablas y gráficas, en fin, explorar a detalle los datos para encontrar condiciones excepcionales y tendencias desde cualquier perspectiva.
          • proyecto dashboard.blackbox.org.ve
        • Capa o vista inferior: en esta capa los usuarios cuentan con reportes detallados y registros de transacciones. Esta información es de utilidad porque permite ir a la causa raíz de un problema.

        De igual forma, las mejores prácticas muestran que existen tres tipos principales de dashboards:

        • Operational dashboards: ayudan a monitorear los procesos operativos clave y son usados principalmente por los empleados y sus supervisores que tratan directamente con clientes y proveedores para gestionar la creación o entrega de productos y servicios. Por lo general, los indicadores en un operational dashboard se actualizan de manera constante, de un día para otro, cada hora o cada minuto, según los requerimientos. Este tipo de dashboard se enfoca principalmente en la parte de monitoreo.
        • Tactical dashboard: permiten el rastreo detallado de procesos y proyectos que son de interés especial para una parte de la organización o un grupo limitado de gente. Los gerentes, coordinadores y analistas de negocio usan este tipo de dashboards para comparar el desempeño de sus áreas o proyectos, realizar planes y pronósticos y evaluar los resultados históricos, entre otras cosas. Por lo general, el contenido de un tactical dashboard se actualiza de manera diaria o semanal, cuya información se presenta en forma detallada y resumida. Este tipo de dashboard se enfoca principalmente en la parte de análisis.
        • Strategic dashboard: facilitan la ejecución del plan estratégico de una organización, en particular de sus objetivos estratégicos. Utilizan una de las metodologías de administración del desempeño más exitosas como es el Balanced Scorecard. Sus principales usuarios son los altos directivos, gerentes de nivel superior y personal de staff. Este tipo de dashboards tiene entre sus principales objetivos la alineación de la organización en torno a sus objetivos estratégicos, la comunicación de la estrategia, el acceso al detalle de las operaciones y la identificación de los generadores clave de valor. Por lo general, el contenido de un strategic dashboard se actualiza una vez al mes o trimestre. Este tipo de dashboard se enfoca principalmente en la parte de administración o gestión.

      Los dashboards bien diseñados dan información que está:

      • Excepcionalmente bien organizada
      • Condensada, con resúmenes y resaltando las excepciones
      • Específica y orientada a las personas que la han de ver y los objetivos a cubrir
      • Mostrada usando mecanismos concisos pero muy claros

Justificación

    • Son herramientas que despliegan la información más importante para el usuario o área, en el tiempo y formato adecuado a sus requerimientos.
    • Cuentan con una interfase de usuario amigable, principalmente vía Web
    • La flexibilidad en su arquitectura y desarrollo permite contar con una variedad de soluciones, dependiendo de si la información es para monitoreo, análisis o administración del desempeño.
    • Ayudan a comunicar la estrategia de la organización mediante el despliegue de objetivos estratégicos, indicadores, metas, procesos y proyectos.
    • Incentivan un mejor desempeño del personal mediante la premisa de "lo que se mide, se hace". Los trabajadores pueden ser motivados a la excelencia en sus funciones, trayendo consigo premios por superar sus objetivos y metas.
    • Facilitan el despliegue de una vista consistente del negocio, al consolidar e integrar información de diversas fuentes usando definiciones, reglas e indicadores comunes.

Objetivo General y Objetivos Específicos

    • Diseñar un dashboard es que entre en una sola página, por lo tanto no podemos malgastar el espacio. La regla de oro en el diseño es hacerlo simple. La elocuencia se consigue simplificando las cosas. El objetivo es mostrar la verdad de la manera más clara y concisa posible Cuando diseña dashboards debemos incluir solamente la información que es necesaria, y condensarla de manera que no disminuya su significado, usando mecanismos de visualización que sean simples pero claros.
    • Los dashboards dicen a los administradors de sistemas que está pasando, y les han de permitir reconocer rápidamente que es lo que necesita su atención.

Implenentación

Django es un framework de desarrollo de aplicaciones web eficiente, versátil y evolucionando de forma dinámica. Cuando Django inicialmente ganó popularidad la configuración recomendada para la ejecución de aplicaciones para Django se basaba en Apache con mod_wsgi, pero Django ha avanzado y en estos días la configuración recomendada es más eficiente y estable. Pero también más complejo e incluye herramientas tales como: Nginx, Gunicorn, virtualenv, supervisord y PostgreSQL. Voy a explicar cómo combinar todos estos componentes par que el servidor ejecute Django en GNUL/Linux Debian 8 ó la version estable hoy.

  • Gunicorn: En producción no vamos a utilizar el servidor de desarrollo. Usaremos un servidor de aplicaciones dedicado llamado gunicorn.

    • Instalamos gunicorn en el entorno virtual de la aplicación:

    (django)webappsusr@blackbox:~$ pip install gunicorn Downloading/unpacking gunicorn Downloading gunicorn-0.17.4.tar.gz (372Kb): 372Kb downloaded Running setup.py egg_info for package gunicorn Installing collected packages: gunicorn Running setup.py install for gunicorn Installing gunicorn_paster script to /var/www/webapps/django/bin Installing gunicorn script to /var/www/webapps/django/bin Installing gunicorn_django script to /var/www/webapps/django/bin Successfully installed gunicorn Cleaning up...

    Ahora que gunicorn está intalado, puede probarse para servir a la aplicación dashboard.blackbox.org.ve ejecutando el siguiente comando:

    (django)webappsusr@blackbox:~$ gunicorn dashboard.blackbox.org.ve.wsgi:application --bind 0.0.0.0:8001

    Ahora debería ser capaz de acceder al servidor Gunicorn de http://dashboard.blackbox.org.ve:8001. Así como está, usando los puertos 8000 y 8001 son a modo desarrollo o prueba. Gunicorn está instalado y listo para servir a nuestra aplicación. Vamos a establecer algunas opciones de configuración para que sea más útil. Estableceré una serie de parámetros, así que vamos a poner a todos en un script bash, que guardo como bin/gunicorn_start.

    (django)webappsusr@blackbox:~$ vim bin/guicorn_start

    #!/bin/bash
    NAME="dashboard.blackbox.org.ve" # Nombre de la aplicación
    DJANGODIR=/var/www/webapps/django/dashboard.blackbox.org.ve # Directorio de la aplicación
    SOCKFILE=/var/www/webapps/django/run/gunicorn.sock # nos comunicaremos con él usando este socket unix
    USER=webappsusr # el usuario que corre la aplicación
    GROUP=webapps # el grupo que corre la aplicación
    NUM_WORKERS=3 # número de procesos trabajando que debe tener spawn Gunicorn
    DJANGO_SETTINGS_MODULE=dashbord.blackbox.org.ve.settings # que archivo de configuracion debe usar Django
    DJANGO_WSGI_MODULE=dashboard.blackbox.org.ve.wsgi # módulo WSGI
    echo "Starting $NAME as `whoami`"
    # Activacion del virtual environment
    cd $DJANGODIR
    source ../bin/activate
    export DJANGO_SETTINGS_MODULE=$DJANGO_SETTINGS_MODULE
    export PYTHONPATH=$DJANGODIR:$PYTHONPATH
    # Crear el directorio Run si no existe
    RUNDIR=$(dirname $SOCKFILE)
    test -d $RUNDIR || mkdir -p $RUNDIR
    # Activar el Gunicorn
    # Programas que deben correr bajo supervisor, no deberán ser demonizados (no utilizar --demonios)
    exec ../bin/gunicorn ${DJANGO_WSGI_MODULE}:application \
    --name $NAME \
    --workers $NUM_WORKERS \
    --user=$USER --group=$GROUP \
    --bind=unix:$SOCKFILE \
    --log-level=debug \

    Establezca el bit de ejecución en el gunicorn_start script:

    $ sudo chmod u+x bin/gunicorn_start

    Probamos gunicorn_start ejecutándolo como el usuario webappsusr

    $ sudo su - webappsusr

    webappsusr@blackbox:~$ bin/gunicorn_start

  • Supervisor: gunicorn_start script está listo y funcionando. Tenemos que asegurarnos de que se inicie automáticamente con el sistema y que se puede reiniciar automáticamente si por alguna razón se cierra inesperadamente. Estas tareas pueden ser fácilmente manejados por un servicio llamado supervisord. La instalación es sencilla:

    • $ sudo aptitude install supervisor

    Cuando se instala Supervisor se puede crear archivos de configuración para que sean auto ejecutados al iniciar el sistema. Todo esto es configurado en el directorio /etc/supervisor/conf.d. para dashborard.blackbox.org.ve crearemos un archivo llamado /etc/supervisor/conf.d/dashboard.blackbox.org.ve.conf con este contenido:

    [program:dashboard.blackbox.org.ve]
    command = /var/www/webapps/django/bin/gunicorn_start ; comando para iniciar la aplicación
    user = webappsusr ; Usuario que corre la aplicación
    stdout_logfile = /var/www/webapps/django/logs/gunicorn_supervisor.log ; Donde los Log's serán depositados
    redirect_stderr = true
    environment=LANG=en_US.UTF-8,LC_ALL=en_US.UTF-8

    Crear el archivo para almacenar los LOG'S de la aplicación:

    webappsusr@blackbox:~$ mkdir -p /var/www/webapps/django/logs/
    webappsusr@blackbox:~$ touch /var/www/webapps/django/logs/gunicorn_supervisor.log

    Después de guardar el archivo de configuración para el programa podemos pedir supervisor de volver a leer los archivos de configuración y de actualización

    $ sudo supervisorctl reread dashboard.blackbox.org.ve: available
    $ sudo supervisorctl update dashboard.blackbox.org.ve: added process group

    La aplicación ahora debe iniciarse automáticamente después de un reinicio del sistema

  • Nginx: Un servidor web de alto rendimiento para la salida de la aplicación y sus archivos estáticos.

    • $ sudo aptitude install nginx
    $ sudo service nginx start

    Crear una configuración de servidor virtual Nginx para Django. Cada servidor virtual Nginx debe ser descrito por un archivo en el directorio /etc/nginx/sites-available. Seleccionaremos los sitios que deseamos habilitar al hacer enlaces simbólicos a la carpeta /etc/nginx/sites-enabled. Crear un nuevo archivo de configuración en nginx para la aplicación Django ejecuta en black box.org.ve en /etc/nginx/sites-available/dashboard.blackbox.org.ve El archivo debe contener las siguientes líneas.

    $ sudo vim /etc/nginx/sites-avalable/dashboard.blackbox.org.ve

    upstream dashboard.blackbox.org.ve {
    # fail_timeout=0 means we always retry an upstream even if it failed
    # to return a good HTTP response (in case the Unicorn master nukes a
    # single worker for timing out).
    server unix:/var/www/webapps/django/run/gunicorn.sock fail_timeout=0;
    }
    server {
    listen 80;
    server_name blackbox.org.ve;
    client_max_body_size 4G;
    access_log /var/www/webapps/django/logs/nginx-access.log;
    error_log /var/www/webapps/django/logs/nginx-error.log;
    location /static/ {
    alias /var/www/webapps/django/static/;
    }
    location /media/ {
    alias /var/www/webapps/django/media/;
    }
    location / {
    # an HTTP header important enough to have its own Wikipedia entry:
    # http://en.wikipedia.org/wiki/X-Forwarded-For
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    # enable this if and only if you use HTTPS, this helps Rack
    # set the proper protocol for doing redirects:
    # proxy_set_header X-Forwarded-Proto https;
    # pass the Host: header from the client right along so redirects
    # can be set properly within the Rack application
    proxy_set_header Host $http_host;
    # we don't want nginx trying to do something clever with
    # redirects, we set the Host: header above already.
    proxy_redirect off;
    # set "proxy_buffering off" *only* for Rainbows! when doing
    # Comet/long-poll stuff. It's also safe to set if you're
    # using only serving fast clients with Unicorn + nginx.
    # Otherwise you _want_ nginx to buffer responses to slow
    # clients, really.
    # proxy_buffering off;
    # Try to serve static files from nginx, no point in making an
    # *application* server like Unicorn/Rainbows! serve static files.
    if (!-f $request_filename) {
    proxy_pass http://dashboard.blackbox.org.ve;
    break;
    }
    }
    # Error pages
    error_page 500 502 503 504 /500.html;
    location = /500.html {
    root /var/www/webapps/django/static/;
    }
    }

    Crear un enlace simbólico en el sites-enabled carpeta:

    $ sudo ln -s /etc/nginx/sites-available/dashboard.blackbox.org.ve /etc/nginx/sites-enabled/dashboard.blackbox.org.ve

    Reiniciar Nginx:

    $ sudo service nginx restart
